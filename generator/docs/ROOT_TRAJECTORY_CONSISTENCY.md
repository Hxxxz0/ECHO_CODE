# Root Trajectory Consistency (RTC)

## 概述

RTC 是一个评价动作生成模型**轨迹一致性**的离线指标，衡量生成动作的根节点移动路径
是否与 GT 动作的路径在形状和覆盖范围上一致。

- **输入**：N 对 (生成动作, GT 动作) 的根节点轨迹
- **输出**：一个标量 ∈ [0, 1]，越高越一致
- **设计原则**：不要求精确位置重合，允许速度差异，关注整体路径形状

## 适用场景

评估 "walk in a circle" 是否真的走了圆形，"walk forward then turn right" 是否走了 L 形，
"walk backward" 是否向后走，等等。

## 前提条件

生成和 GT 的轨迹已经过预处理（符合 Robot-Skeleton Motion Representation 流程）：

- 起始位置：根节点 XY = (0, 0)
- 初始朝向：面向 +X 方向
- 落地归一化：最低点 Z = 0
- 坐标系：Z-up（XY 平面为地面）

因此两条轨迹**天然对齐了起点和初始方向**，无需额外对齐。

## 数据提取

本实现使用 **38 维特征**（velocity-based representation），通过积分 `root_vel_xy` 恢复根轨迹，
而非直接从 NPZ 的 body_pos_w 读取。这样 GT 与生成数据保持相同的数据流和预处理一致性。

```python
# 方案 A：从 NPZ 数据中提取（若使用原始 body_pos_w）
# body_pos_w: (T, 30, 3)，root_idx = 0
root_traj_gen = body_pos_w_gen[:, 0, :2]   # (T_gen, 2)
root_traj_gt  = body_pos_w_gt[:, 0, :2]    # (T_gt, 2)

# 方案 B：从 38 维特征积分恢复（本实现）
# 38D: [joint_pos(29), root_vel_xy(2), root_z(1), root_rot_6d(6)]
# root_traj = recover_root_xy_from_velocity(root_vel_xy, root_rot_quat)
```

**评估配对**：生成动作与 GT 必须一一对应（相同 text prompt）。本实现通过 **caption 匹配**
建立配对，避免因 DataLoader 的 shuffle 导致错误配对。

## 计算公式

### 两个子分数

RTC 由两个互补的子分数组成：

| 子分数 | 权重 | 衡量内容 |
|--------|------|---------|
| **Shape Score (S_shape)** | 0.7 | 路径形状是否一致（圆形 vs 直线 vs L形...） |
| **Extent Score (S_extent)** | 0.3 | 移动总距离是否一致（有没有走完/走过头） |

---

### Sub-score 1: Shape Score (S_shape)

**核心思想**：用**弧长参数化**消除速度差异，只比较路径形状。

> 弧长参数化：将"沿路径走了百分之几"作为比较基准，而非"过了多少时间"。
> 这样，即使生成动作走得快一倍，只要路径形状一致，就能得到高分。

**步骤**：

```
1. 计算两条轨迹的累计弧长:
   s_gen[t] = Σ_{i=0}^{t-1} ||P_gen[i+1] - P_gen[i]||₂
   s_gt[t]  = Σ_{i=0}^{t-1} ||P_gt[i+1] - P_gt[i]||₂

2. 归一化弧长到 [0, 1]:
   s_gen_norm = s_gen / L_gen    (L_gen = s_gen[-1]，总弧长)
   s_gt_norm  = s_gt  / L_gt

3. 按弧长均匀重采样为 K 个点（如 K=50）:
   fractions = [0, 1/K, 2/K, ..., 1]
   Q_gen = interpolate(P_gen, s_gen_norm, fractions)   # (K, 2)
   Q_gt  = interpolate(P_gt,  s_gt_norm,  fractions)   # (K, 2)

   含义：Q_gen[i] 和 Q_gt[i] 都是"沿路径走了 i/K 比例处"的位置

4. 逐点计算欧式距离并取均值:
   d = (1/K) × Σ_{i=0}^{K-1} ||Q_gen[i] - Q_gt[i]||₂

5. 归一化（以两条轨迹的平均总弧长为基准）:
   d_norm = d / ((L_gen + L_gt) / 2)

6. 映射为分数:
   S_shape = exp(-(d_norm / σ_shape)²)
   σ_shape = 0.35  （实现中采用，较原 0.2 更宽松以提升分数可读性）
```

**直觉**（σ_shape = 0.35）：
- d_norm = 0（完美重合）→ S_shape = 1.0
- 平均偏差 = 总路径长度的 10% → S_shape ≈ 0.92
- 平均偏差 = 总路径长度的 20% → S_shape ≈ 0.70
- 平均偏差 = 总路径长度的 35% → S_shape ≈ 0.37
- 形状完全不同（直线 vs 圆）→ S_shape ≈ 0.0

**为什么用弧长而非时间**：
- "walk in a circle"：GT 走 4s，Gen 走 6s 但路径相同 → 时间对齐会惩罚速度差异，
  弧长对齐则只看形状，给满分。
- 这符合"不要非常严格"的要求。

---

### Sub-score 2: Extent Score (S_extent)

**核心思想**：检查"有没有走完"。如果 GT 走了一个完整的圆，但 Gen 只走了半圆，
即使形状正确（半圆 ⊂ 圆），覆盖程度不够也应该扣分。

**步骤**：

```
1. 计算两条轨迹的总弧长（即总行走距离）:
   L_gen, L_gt

2. 计算弧长偏差:
   Δ = |L_gen - L_gt| / max(L_gt, ε)

3. 映射为分数:
   S_extent = exp(-(Δ / σ_extent)²)
   σ_extent = 0.8  （实现中采用，较原 0.5 更宽松以提升分数可读性）
```

**直觉**（σ_extent = 0.8）：
- GT 走 3m，Gen 走 3m → S_extent = 1.0
- GT 走 3m，Gen 走 2m（少走 33%）→ S_extent ≈ 0.84
- GT 走 3m，Gen 走 1.5m（少走 50%）→ S_extent ≈ 0.68
- GT 走 3m，Gen 走 1m（少走 67%）→ S_extent ≈ 0.42
- GT 走 3m，Gen 原地不动 → S_extent ≈ 0.02

σ_extent = 0.8 表示 **容许更大距离偏差**（约 ±80% 对应分数 0.37），体现"不要非常严格"。

---

### 合成：RTC

```
RTC_i = S_shape ^ 0.7  ×  S_extent ^ 0.3
```

加权几何平均，形状权重更高（路径对不对比走没走完更重要）。

### 测试集总分

```
RTC = (1/N) × Σ RTC_i
```

## 边界情况处理

| 情况 | 处理 |
|------|------|
| 双方都是静态（L_gen < ε 且 L_gt < ε） | RTC_i = 1.0（都不动 = 一致） |
| 仅一方静态（如 L_gen < ε 但 L_gt > ε） | S_extent ≈ 0 → RTC_i ≈ 0 |
| 序列极短（< 3 帧） | K = min(K, T_min)，保证有效采样 |

## 示例

### 例1：走圆形（接近完美）
```
GT:  半径 1.5m 的完整圆，L_gt ≈ 9.4m
Gen: 半径 1.6m 的完整圆，L_gen ≈ 10.1m

Shape: 平均偏差 ≈ 0.1m，d_norm ≈ 0.1/9.75 ≈ 0.010 → S_shape ≈ 0.998
Extent: Δ = |10.1-9.4|/9.4 ≈ 0.074 → S_extent ≈ 0.978
RTC = 0.998^0.7 × 0.978^0.3 ≈ 0.992 ✓ 优秀
```

### 例2：走圆形但只走了半圆
```
GT:  完整圆，L_gt ≈ 9.4m
Gen: 半圆，L_gen ≈ 4.7m

Shape: 弧长参数化后，Gen 的后半段点还在半圆末端，而 GT 的点继续绕回，
       平均偏差很大 → d_norm ≈ 0.5 → S_shape ≈ 0.002
Extent: Δ = 0.5 → S_extent ≈ 0.37
RTC = 0.002^0.7 × 0.37^0.3 ≈ 0.006 × 0.73 ≈ 0.004 ✗ 危险
```

### 例3：走直线 vs GT 走圆形
```
GT:  圆形路径，L_gt ≈ 9.4m
Gen: 直线前进，L_gen ≈ 9.4m

Shape: 弧长参数化后，直线进程与圆形进程完全不同，
       平均偏差 ≈ 路径长度的 60% → S_shape ≈ 0.0
Extent: Δ ≈ 0 → S_extent ≈ 1.0
RTC = 0.0^0.7 × 1.0^0.3 ≈ 0.0 ✗ 完全不一致
```

### 例4：走直线但速度不同（应得高分）
```
GT:  直线向前 3m，4s
Gen: 直线向前 3m，6s（更慢）

Shape: 弧长参数化消除速度差，两条直线完全重合 → S_shape ≈ 1.0
Extent: Δ ≈ 0 → S_extent ≈ 1.0
RTC ≈ 1.0 ✓ 完美（速度差异被容许）
```

## 输出解读

| 分数区间 | 含义 |
|----------|------|
| ≥ 0.90 | 优秀：轨迹形状和覆盖都高度吻合 |
| 0.70 ~ 0.90 | 良好：轨迹大致正确，存在轻微偏差 |
| 0.40 ~ 0.70 | 中等：轨迹有明显偏差但方向基本正确 |
| < 0.40 | 差：轨迹形状或覆盖范围严重不匹配 |

## 与 MSS 的关系

- **MSS**（Motion Safety Score）：关注动作本身是否安全（关节角度/速度/加速度）
- **RTC**（Root Trajectory Consistency）：关注动作的全局移动是否正确

两个指标互补：一个动作可以 MSS 高（关节安全）但 RTC 低（走错了路线），反之亦然。
评测时同时报告两个分数：

```
模型评估结果:
  MSS = 0.87 （安全性：良好）
  RTC = 0.92 （轨迹一致性：优秀）
```

## 设计依据

1. **弧长参数化消除速度差异**。生成模型的速度控制可能不精确，但只要路径形状对了就应该
   得到认可。这直接回应了"不要非常严格"的要求。

2. **Shape + Extent 双维度**。只看形状会漏掉"没走完"的情况（半圆 vs 全圆形状都是弧形），
   加了 Extent 就能捕捉这个问题。

3. **权重 0.7 : 0.3**。路径形状比覆盖范围更重要——走对了方向但没完全走完，
   比走了足够远但方向错误要好。

4. **σ 值的选择**。实现中采用 σ_shape=0.35、σ_extent=0.8（较文档原 0.2/0.5 更宽松），
   以提升分数可读性和区分度。σ 越大，对偏差越宽容，分数越高。

5. **起点和朝向已对齐**。数据预处理保证了两条轨迹从同一点出发、朝同一方向，
   因此无需 Procrustes 对齐等额外步骤，直接比较即可。
